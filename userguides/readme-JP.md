# 📕 Memory Books (SillyTavern 拡張機能)

自動的かつ構造化された信頼性の高いメモリ作成を行うための、次世代 SillyTavern 拡張機能です。チャット内のシーンをマークし、AI を使って JSON ベースの要約を生成し、それらをロアブックのエントリとして保存します。グループチャット、高度なプロファイル管理、サイドプロンプト/トラッカー、および多段階のメモリ統合をサポートしています。

### ❓ 用語集
- Scene（シーン）→ Memory（メモリ）
- Many Memories（複数のメモリ）→ Summary / Consolidation（要約 / 統合）
- Always-On（常時稼働）→ Side Prompt (Tracker)（サイドプロンプト / トラッカー）

## 📋 必須要件 (Prerequisites)

- **SillyTavern:** 1.14.0 以上（最新版を推奨）
- **シーン選択:** 開始マーカーと終了マーカー（開始 < 終了）が設定されている必要があります。
- **Chat Completion サポート:** OpenAI、Claude、Anthropic、OpenRouter、またはその他の Chat Completion API を完全にサポートしています。
- **Text Completion サポート:** Kobold、TextGen などの Text Completion API は、Chat Completion（OpenAI 互換）API エンドポイント経由で接続されている場合にサポートされます。下記の KoboldCpp のヒントに従って Chat Completion API 接続を設定することをお勧めします（Ollama やその他のソフトウェアを使用している場合は必要に応じて調整してください）。その後、STMB プロファイルを設定し、`Custom`（推奨）または完全な手動設定（Custom が失敗する場合、またはカスタム接続が複数ある場合のみ）を使用してください。

**注:** Text Completion を使用する場合は、Chat Completion プリセットが必須です。

### KoboldCpp で 📕 ST Memory Books を使用するためのヒント

ST で以下のように設定してください（STMB が動作することを確認した後で Text Completion に戻せます）。

- Chat Completion API
- Custom chat completion source
- エンドポイント: `http://localhost:5001/v1`（`127.0.0.1:5000/v1` も使用できます）
- `custom API key` には何かを入力してください（内容は何でも構いませんが、ST では必須です）。
- モデル ID は `koboldcpp/modelname` にする必要があります（モデル名に `.gguf` を入れないでください！）。
- Chat Completion プリセットをダウンロードしてインポートしてください（どれでも構いません）。Chat Completion プリセットが存在するだけで、「not supported」系のエラーを避けられます。
- Chat Completion プリセットの最大応答長を少なくとも 2048 に変更してください。4096 を推奨します（小さいと途中で切れるリスクがあります）。

### Llama.cpp で 📕 ST Memory Books を使用するためのヒント

Kobold と同様に、ST で _Chat Completion API_ として以下を設定してください（STMB が動作することを確認した後で元の設定に戻せます）。

- Chat Completion API 用の新しい接続プロファイルを作成します。
- Completion Source: `Custom (Open-AI Compatible)`
- Endpoint URL: Docker で ST を実行している場合は `http://host.docker.internal:8080/v1`、それ以外は `http://localhost:8080/v1`
- Custom API key: 何かを入力してください（ST では必須です）。
- Model ID: `llama2-7b-chat.gguf`（または使用しているモデル。llama.cpp で複数モデルを実行していない場合は何でも構いません）
- Prompt post-processing: none

Llama.cpp を起動する場合、起動を簡単にするために、以下のような内容をシェルスクリプトまたは bat ファイルに入れることをお勧めします。

```sh
llama-server -m <model-path> -c <context-size> --port 8080
```

## 💡 推奨されるグローバル World Info/ロアブック有効化設定

- **Match Whole Words:** チェックなし（false）のままにしてください。
- **Scan Depth:** 高いほど良いです（私の設定は 8 です）。
- **Max Recursion Steps:** 2（一般的な推奨値であり、必須ではありません）。
- **Context %:** 80%（100,000 トークンのコンテキストウィンドウに基づく）- チャット履歴やボットが極端に重くないことを前提としています。
- 追加の注意: メモリロアブックが唯一のロアブックである場合は、STMB プロファイルで `Delay until recursion` が無効になっていることを確認してください。有効だとメモリがトリガーされません。

- シーンの最後のメッセージで ◄ をクリックします。

- プロンプトが表示された場合は、設定（プロファイル、コンテキスト、API/モデル）を確認します。
- AI による生成と自動ロアブックエントリ作成を待ちます。

## 🧩 メモリの種類: シーン vs 要約

📕 Memory Books は、**シーンメモリ** と **多段階の要約統合** をサポートしています。それぞれ、異なる種類の連続性を扱うために設計されています。

### 🎬 シーンメモリ (デフォルト)

シーンメモリは、特定のメッセージ範囲で **何が起こったか** を記録します。

- 明示的なシーン選択（► ◄）に基づきます。
- 瞬間ごとの回想に適しています。
- 会話、行動、直後の結果を保持します。
- 頻繁に使用するのに向いています。

これは標準的で、最も一般的に使用されるメモリタイプです。

---

### 🌈 要約統合

要約統合は、複数のメモリまたは要約にまたがって **時間の経過とともに何が変わったか** を記録します。

単一シーンを要約するのではなく、統合要約は以下に焦点を当てます。

- キャラクターの成長と関係性の変化
- 長期的な目標、緊張、解決
- 感情の軌跡と物語の方向性
- 安定して残すべき永続的な状態変化

最初の統合階層は **Arc** で、シーンメモリから作られます。長く続く物語向けに、さらに上位の階層もサポートされています。

- Arc
- Chapter
- Book
- Legend
- Series
- Epic

> 💡 これはシーンログではなく、*これまでのあらすじ* と考えてください。

#### 統合要約を使うタイミング

- 大きな関係性の変化があった後
- 章やアークの終わり
- 動機、信頼、力関係が変化したとき
- 物語の新しいフェーズを始める前

#### 仕組み

- 統合要約は、生のチャット本文からではなく、既存の STMB メモリ/要約から生成されます。
- **Consolidate Memories** ツールでは、対象の要約階層と元になるエントリを選択できます。
- STMB は、選択した要約階層を監視し、その階層が保存済みの最小対象数に達したときに yes/later の確認を表示できます。
- より高レベルの要約に役割を引き継がせたい場合、統合後に元のエントリを無効化できます。
- AI の要約応答が失敗した場合は、UI で確認・修正してからコミットを再試行できます。

これにより、以下の効果があります。

- トークン使用量の削減
- 長いチャットでの物語の連続性向上

## 📝 メモリ生成

### **JSON のみ出力**

すべてのプロンプトとプリセットは、AI に有効な JSON のみを返すよう指示する **必要があります**。例：

```json
{
  "title": "短いシーンタイトル",
  "content": "シーンの詳細な要約...",
  "keywords": ["キーワード1", "キーワード2"]
}
```

**応答に他のテキストを含めることは許可されていません。**

### **組み込みプリセット**

1. **Summary:** 詳細なビートごとの要約。
2. **Summarize:** タイムライン、ビート、相互作用、結果のための Markdown ヘッダー。
3. **Synopsis:** 包括的で構造化された Markdown。
4. **Sum Up:** タイムライン付きの簡潔なビート要約。
5. **Minimal:** 1〜2 文の要約。
6. **Northgate:** 創作向けの文学的な要約スタイル。
7. **Aelemar:** プロット要点とキャラクターの記憶に重点を置きます。
8. **Comprehensive:** キーワード抽出を改善したシノプシス型要約。

### **カスタムプロンプト**

- 独自に作成できますが、上記のように有効な JSON を返す **必要があります**。

## 📚 ロアブック統合

- **自動エントリ作成:** 新しいメモリは、すべてのメタデータを含むエントリとして保存されます。
- **フラグベースの検出:** `stmemorybooks` フラグを持つエントリのみがメモリとして認識されます。
- **自動ナンバリング:** 複数の形式（`[000]`, `(000)`, `{000}`, `#000`）をサポートする、連番のゼロ埋めナンバリングです。
- **手動/自動順序:** プロファイルごとの挿入順序設定。
- **エディタの更新:** メモリを追加した後、必要に応じてロアブックエディタを自動更新します。

> **既存のメモリは変換が必要です！**
> [`Lorebook Converter`](../resources/lorebookconverter.html) を使用して、`stmemorybooks` フラグと必須フィールドを追加してください。

## 🆕 スラッシュコマンド

- `/creatememory` - マーク済みシーンからメモリを作成します。
- `/scenememory X-Y` - シーン範囲を設定してメモリを作成します（例: `/scenememory 10-15`）。
- `/nextmemory` - 前回のメモリの終端から現在のメッセージまでを使ってメモリを作成します。
- `/stmb-catchup interval:x start:y end:y` - 既存の長いチャットを、指定したメッセージ範囲ごとに interval サイズのチャンクへ分割して、キャッチアップ用メモリを作成します。
- `/sideprompt "Name" {{macro}}="value" [X-Y]` - サイドプロンプトを実行します（`{{macro}}` は任意です）。
- `/sideprompt-on "Name" | all` - 名前指定または `all` で Side Prompt を有効化します。
- `/sideprompt-off "Name" | all` - 名前指定または `all` で Side Prompt を無効化します。
- `/stmb-highest` - このチャットで処理済みメモリの最も高い message id を返します。
- `/stmb-set-highest <N|none>` - このチャットの処理済み最高 message id を手動で設定します。
- `/stmb-stop` - 進行中のすべての STMB 生成を全体で停止します（緊急停止）。

### `/stmb-catchup`

既存の長いチャットを STMB メモリに変換したいときに `/stmb-catchup` を使用します。構文: `/stmb-catchup interval:x start:y end:y`

例: `/stmb-catchup interval:30 start:0 end:300`

## 🧭 動作モード

### **自動モード (デフォルト)**

- **動作:** 現在のチャットにバインドされているロアブックを自動的に使用します。
- **推奨:** シンプルさとスピード重視の場合。ほとんどのユーザーはここから始めるべきです。
- **使用法:** キャラクターまたはグループチャットの `Chat Lorebooks` ドロップダウンでロアブックが選択されていることを確認してください。

![チャットロアブックのバインド例](https://github.com/aikohanasaki/imagehost/blob/main/STMemoryBooks/chatlorebook.png)

### **ロアブック自動作成モード**

- **動作:** ロアブックが存在しない場合、カスタム命名テンプレートを使用して新しいロアブックを自動的に作成し、バインドします。
- **推奨:** 新規ユーザーや素早いセットアップに適しています。ワンクリックでのロアブック作成に向いています。
- **使用法:**
  1. 拡張機能の設定で `Auto-create lorebook if none exists` を有効にします。
  2. 命名テンプレートを設定します（デフォルト: `LTM - {{char}} - {{chat}}`）。
  3. バインドされたロアブックがない状態でメモリを作成すると、ロアブックが自動的に作成・バインドされます。
- **テンプレートプレースホルダー:** `{{char}}`（キャラクター名）、`{{user}}`（あなたの名前）、`{{chat}}`（チャット ID）
- **スマートナンバリング:** 重複する名前が存在する場合、自動的に番号（2, 3, 4...）を追加します。
- **注:** 手動ロアブックモードとは同時に使用できません。

### **手動ロアブックモード**

- **動作:** メインのチャットバインドロアブックを無視して、チャットごとにメモリ用の別ロアブックを選択できます。
- **推奨:** メモリを特定の別ロアブックに保存したい上級ユーザー向けです。
- **使用法:**
  1. 拡張機能の設定で `Enable Manual Lorebook Mode` を有効にします。
  2. チャットで初めてメモリを作成するときに、ロアブックを選択するよう求められます。
  3. この選択は、クリアするか自動モードに戻すまで、そのチャットに保存されます。
- **注:** ロアブック自動作成モードとは同時に使用できません。

## 🎡 トラッカーとサイドプロンプト

サイドプロンプトはトラッカーのように使用でき、メモリロアブック内に個別のサイドプロンプトエントリを作成します。サイドプロンプトを使うと、過去のイベントだけでなく、**進行中の状態** を追跡できます。例えば：

- 💰 インベントリとリソース（「ユーザーはどのアイテムを持っているか？」）
- ❤️ 関係ステータス（「X は Y についてどう感じているか？」）
- 📊 キャラクター統計（「現在の体力、スキル、評判」）
- 🎯 クエスト進行状況（「どの目標がアクティブか？」）
- 🌍 世界の状態（「設定で何が変わったか？」）

#### **アクセス:** Memory Books 設定から `🎡 Trackers & Side Prompts` をクリックします。

#### **機能:**

- すべてのサイドプロンプトを表示します。
- 新しいプロンプトを作成したり複製したりして、異なるプロンプトスタイルを試せます。
- 組み込みを含む任意のプリセットを編集または削除できます。
- バックアップや共有のためにプリセットを JSON ファイルとしてエクスポート/インポートできます。
- テンプレートに応じて、手動または自動で実行できます。
- サイドプロンプトの `Prompt`、`Response Format`、`Title`、`{{keyword}}` フィールドで、`{{user}}` や `{{char}}` のような標準 SillyTavern マクロ/プレースホルダーを使用できます。
- `{{npc name}}` のようなカスタムマクロ/プレースホルダーを使用できます（`/sideprompt` 実行時に値を指定する必要があります）。

#### **使用のヒント:**

- 新しいプロンプトを作成する場合、互換性を高めるために組み込みからコピーできます。
- サイドプロンプトは JSON を返す必要はありません。プレーンテキストを返せます。
- サイドプロンプトは更新/上書きされます。これは、連番で保存されるメモリとの違いです。
- 手動構文は `/sideprompt "Name" {{macro}}="value" [X-Y]` です。
- スラッシュコマンドのオートコンプリートでサイドプロンプトを選ぶと、STMB はそのテンプレートに必要なランタイムマクロを提案します。
- カスタムランタイムマクロ（ST 標準ではないもの）を含むサイドプロンプトは手動専用です。STMB は保存/インポート時にそれらのテンプレートから `On Interval` と `On After Memory` を無効化し、その際に警告を表示します。
- 追加のサイドプロンプトテンプレートライブラリは [JSON ファイル](../resources/SidePromptTemplateLibrary.json) として用意されています。インポートするだけで使用できます。

## 🧠 高度なカスタマイズのための Regex 統合

- **テキスト処理の完全制御:** Memory Books は SillyTavern の **Regex** 拡張機能と統合され、次の 2 つの重要な段階で強力なテキスト変換を適用できます。
  1. **プロンプト生成:** **User Input** 配置を対象にした regex スクリプトを作成することで、AI に送信されるプロンプトを自動的に変更します。
  2. **応答解析:** **AI Output** 配置を対象にすることで、保存前に AI の生応答をクリーンアップ、再フォーマット、標準化します。
- **複数選択のサポート:** 送信前処理と受信後処理のそれぞれについて、複数のスクリプトを選択できます。
- **仕組み:** STMB で `Use regex (advanced)` を有効にし、`📐 Configure regex…` をクリックして、AI への送信前および解析/保存前に実行するスクリプトを選択します。
- **重要:** Regex の選択は STMB 側で制御されます。ここで選択されたスクリプトは、Regex 拡張機能側で現在無効になっていても **実行されます**。

## 🏷️ タイトルのフォーマット

強力なテンプレートシステムを使用して、ロアブックエントリのタイトルをカスタマイズできます。

# FAQ

### メモリ用に別のロアブックを作成すべきですか？それとも他の用途ですでに使用しているものと同じロアブックを使用できますか？

メモリ用ロアブックは別のブックにすることをお勧めします。他のエントリと比べてメモリを整理しやすくなります。たとえば、グループチャットへ追加する、別のチャットで使用する、または STLO を使って個別のロアブック予算を設定する場合に便利です。

### ベクトルを実行する必要がありますか？

使用しても構いませんが、必須ではありません。Vectors 拡張機能を使用していない場合（私は使用していません）、キーワード経由で動作します。これは自動化されているため、どのキーワードを使うかを考える必要はありません。

### Memory Books が唯一のロアブックである場合、「Delay until recursion」を使用すべきですか？

いいえ。他の World Info やロアブックがない場合、`Delay until recursion` を選択すると最初のループがトリガーされず、何もアクティブにならない可能性があります。Memory Books が唯一のロアブックである場合は、`Delay until recursion` を無効にするか、少なくとも 1 つの追加 World Info/ロアブックを設定してください。

### なぜ AI がエントリを見ていないのですか？

まず、エントリが実際に送信されているか確認する必要があります。私はその確認に [WorldInfo-Info](https://github.com/aikohanasaki/SillyTavern-WorldInfoInfo) を使うのが好きです。

エントリがトリガーされ、AI に送信されている場合は、OOC で AI に強めに指摘するとよいかもしれません。例: `[OOC: WHY are you not using the information you were given? Specifically: (whatever it was)]` 😁

# Troubleshooting

- **拡張機能メニューに Memory Books が見つかりません！**
  設定は拡張機能メニュー（入力欄の左側にある魔法の杖 🪄）にあります。「Memory Books」を探してください。

![STMB 設定の場所](https://github.com/aikohanasaki/imagehost/blob/main/STMemoryBooks/menu.png)

- **ロアブックが利用できないか、選択されていない:**
  - 手動モードの場合、プロンプトが表示されたらロアブックを選択してください。
  - 自動モードの場合、ロアブックをチャットにバインドしてください。
  - または、自動作成のために `Auto-create lorebook if none exists` を有効にしてください。

- **Lorebook Validation Error:**
  - 以前バインドされていたロアブックを削除した可能性があります。新しいロアブックをバインドしてください（空でも構いません）。

- **シーンが選択されていない:**
  - 開始（►）ポイントと終了（◄）ポイントの両方をマークしてください。

- **シーンが既存のメモリと重複している:**
  - 別の範囲を選択するか、設定で `Allow Scene Overlap` を有効にしてください。

![シーン重複警告](https://github.com/aikohanasaki/imagehost/blob/main/STMemoryBooks/overlap.png)

- **AI が有効なメモリの生成に失敗した:**
  - JSON 出力をサポートするモデルを使用してください。
  - プロンプトとモデル設定を確認してください。

- **Token warning threshold を超えた:**
  - より小さなシーンを使用するか、しきい値を上げてください。

- **山括弧ボタンがない:**
  - 拡張機能が読み込まれるのを待つか、更新してください。

- **キャラクターデータが利用できない:**
  - チャット/グループが完全に読み込まれるのを待ってください。

## 👨‍💻 開発者向け

### 拡張機能のビルド

この拡張機能はビルドに Bun を使用します。ビルド処理では、ソースファイルを minify して bundle します。

```sh
# 拡張機能をビルド
bun run build
```

### Git フック

このプロジェクトには、拡張機能を自動的にビルドし、ビルド成果物をコミットに含める pre-commit フックが含まれています。これにより、ビルド済みファイルが常にソースコードと同期されます。

**Git フックをインストールするには:**

```sh
bun run install-hooks
```

このフックは次の処理を行います。

- 各コミット前に `bun run build` を実行します。
- ビルド成果物をコミットに追加します。
- ビルドが失敗した場合、コミットを中止します。
